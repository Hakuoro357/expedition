# Solitaire: Expedition — Tech Spec

## Цель

Зафиксировать технический стек и минимальную архитектуру `v1.0`, чтобы можно было быстро начать разработку без лишних развилок.

## Главный принцип

Для первой игры не нужен "идеальный" стек. Нужен стек, который:

- быстро поднимается;
- стабильно работает в Web;
- нормально живет в Яндекс Играх;
- не требует тяжелой инфраструктуры;
- позволяет выпустить MVP за `2 недели`.

## Выбранный технический подход

### Рекомендуемый стек

- `TypeScript`
- `Phaser 3`
- `Vite`
- `Yandex Games SDK`
- `localStorage` + платформенные сохранения Яндекс Игр

## Почему именно так

### Почему `TypeScript`

- меньше глупых ошибок на старте;
- проще держать модель карт, состояния партии и прогресса;
- легче масштабировать после первого релиза.

### Почему `Phaser 3`

- быстрый старт для 2D Web-игры;
- достаточно возможностей для drag and drop, сцен, анимаций и адаптивного UI;
- не нужно писать рендеринг и ввод с нуля;
- хорош для карточной логики и простой меты.

### Почему не "чистый Canvas" или "чистый DOM"

- выше риск вязнуть в низкоуровневой реализации;
- больше технического шума ради MVP;
- меньше скорости на старте.

### Почему не Unity

- лишний вес и лишняя сложность для первого web-релиза;
- больше риск по размеру билда и времени сборки;
- для карточной игры это избыточно.

## Технические цели `v1.0`

- быстрый старт партии;
- стабильная работа на мобильном браузере;
- корректный responsive layout;
- сохранение прогресса;
- поддержка rewarded ads;
- локализация `RU + EN`;
- простая и читаемая структура проекта.

## Архитектурный подход

Игра должна быть разделена на 3 слоя:

1. `Game Logic`
2. `Presentation`
3. `Platform Services`

## 1. Game Logic

Это чистая логика игры без привязки к Phaser UI.

### Что входит

- модель карты;
- модель колоды;
- правила ходов;
- проверка победы;
- проверка доступных ходов;
- генерация / загрузка расклада;
- состояние партии;
- состояние прогресса Adventure.

### Почему это важно

Если логика отделена от UI:

- проще тестировать;
- проще чинить баги;
- проще добавлять новые режимы потом.

## 2. Presentation

Это слой Phaser-сцен и визуального интерфейса.

### Что входит

- экран загрузки;
- карта экспедиции;
- экран партии;
- экран награды;
- дневник;
- настройки;
- анимации и HUD.

### Правило

UI не должен хранить бизнес-логику правил пасьянса.

## 3. Platform Services

Это слой интеграций.

### Что входит

- SDK Яндекс Игр;
- сохранения;
- rewarded ads;
- локализация;
- аналитика;
- конфиг приложения.

## Модель данных

### Card

Минимальная структура:

- `id`
- `suit`
- `rank`
- `color`
- `faceUp`

### Pile

Типы:

- `stock`
- `waste`
- `tableau`
- `foundation`

### GameState

Минимум:

- массив piles;
- история ходов для `undo`;
- текущий статус партии;
- число подсказок / наградных действий;
- идентификатор расклада;
- флаг daily / adventure.

### ProgressState

Минимум:

- текущая глава;
- открытые точки;
- пройденные точки;
- монеты;
- найденные артефакты;
- серия входов;
- статус daily на текущую дату;
- выбранные косметические настройки.

## Сцены `v1.0`

### `BootScene`

Задачи:

- загрузка ассетов;
- инициализация сервисов;
- восстановление сохранения;
- переход в нужную сцену.

### `MapScene`

Задачи:

- показать карту экспедиции;
- выбрать точку маршрута;
- открыть дневник, daily, настройки;
- запускать уровень.

### `GameScene`

Задачи:

- отрисовать карточное поле;
- принимать ввод;
- синхронизироваться с game state;
- обрабатывать победу и поражение.

### `RewardScene`

Задачи:

- показать награду;
- предложить rewarded-бонус;
- вернуть игрока на карту.

### `DiaryScene`

Задачи:

- показывать найденные артефакты и прогресс.

### `SettingsScene`

Задачи:

- язык;
- звук;
- музыка;
- служебные настройки.

## Игровая логика

### Источник истины

Единственный источник истины для партии — объект `GameState`.

UI только:

- отправляет действия;
- получает новое состояние;
- перерисовывает экран.

### Формат действий

Нужны явные команды:

- `drawCard`
- `moveCard`
- `moveStack`
- `autoMove`
- `undoMove`
- `restartDeal`
- `requestHint`

Это уменьшает хаос в логике.

## Сохранения

### Уровни сохранения

1. `local save`
2. `cloud/platform save`

### Что сохраняем

- прогресс Adventure;
- дневник;
- валюту;
- daily-статус;
- текущую незаконченную партию;
- настройки языка и звука.

### Подход

Сначала сохраняем локально.
Потом синхронизируем с платформой, если SDK доступен и пользователь авторизован.

### Защита от проблем

- сохранение должно быть версионированным;
- битое сохранение не должно ломать запуск игры;
- нужен fallback на дефолтный state.

## Интеграция SDK Яндекс Игр

### Что нужно в `v1.0`

- инициализация SDK;
- доступ к профилю / авторизации при наличии;
- сохранения;
- rewarded ads.

### Техническое правило

Код игры не должен быть жестко завязан на прямые вызовы SDK в каждой сцене.

Нужен адаптер:

- `YandexSdkService`
- `AdsService`
- `SaveService`

## Монетизация

### Только rewarded ads в `v1.0`

Сценарии:

- бонус за победу;
- дополнительный `hint`;
- дополнительный `undo`, если это остается в дизайне.

### Технические требования

- после отказа реклама не ломает сценарий;
- после ошибки реклама не блокирует UI;
- при отсутствии SDK игра все равно должна быть играбельной.

## Локализация

### Языки `v1.0`

- `ru`
- `en`

### Подход

- все тексты только через ключи;
- строки хранятся отдельно;
- язык определяется из платформы или настроек;
- есть ручное переключение в настройках.

## Аналитика

### Минимальные события

- `session_start`
- `map_open`
- `deal_start`
- `deal_win`
- `deal_fail`
- `hint_used`
- `undo_used`
- `rewarded_offer_shown`
- `rewarded_offer_accept`
- `rewarded_offer_complete`
- `daily_open`
- `daily_complete`
- `chapter_complete`

### Подход

Слой аналитики должен быть отдельным сервисом, даже если на старте события пишутся в консоль или временный адаптер.

## Адаптивность

### Целевые форматы

- мобильный portrait — главный приоритет;
- мобильный landscape — поддержка без идеального UX;
- desktop — поддержка обязательна.

### Правила

- поле должно влезать без потери читаемости;
- кнопки должны быть доступны большим пальцем;
- мета-экраны не должны ломаться на узких экранах.

## Производительность

### Цели

- быстрая загрузка;
- плавный drag and drop;
- минимум тяжелых эффектов;
- умеренный вес ассетов.

### Что избегать

- больших нежатых изображений;
- тяжелых частиц;
- сложных анимационных цепочек;
- постоянных перерасчетов layout без нужды.

## Тестирование

### Что обязательно проверять руками

- старт новой игры;
- возврат в незавершенную партию;
- победа;
- проигрыш и рестарт;
- daily;
- сохранение и восстановление;
- rewarded ads;
- переключение языка;
- мобильный layout;
- desktop layout.

### Что желательно покрыть кодом

- правила допустимых ходов;
- генерацию стартовой раскладки;
- проверку победы;
- восстановление состояния партии.

## Error handling

### Принцип

Ошибка платформы не должна ломать core gameplay.

### Примеры fallback-поведения

- нет SDK -> запускаем локальный режим;
- нет рекламы -> скрываем rewarded-опцию;
- битое сохранение -> создаем новое и логируем ошибку;
- недоступен cloud save -> работаем на local save.

## Сборка и деплой

### Build target

- production web build для Яндекс Игр.

### На релиз нужно

- production build;
- проверка размера бандла;
- smoke test после сборки;
- ручной проход ключевых сценариев.

## Definition of Done для технической части

- проект собирается без ошибок;
- базовый `Klondike` стабильно работает;
- прогресс сохраняется и восстанавливается;
- rewarded ads подключены или корректно скрыты в fallback;
- `RU + EN` работают;
- игра не падает без платформенных сервисов;
- основная навигация между сценами завершена.

## Что сознательно не усложняем в `v1.0`

- глобальный state manager уровня enterprise;
- тяжелую DI-систему;
- полноценный backend;
- сложную систему remote config;
- онлайн-профили и мультиплеер;
- гибкую live-ops платформу.

## Решения, которые можно пересмотреть после релиза

- добавить второй игровой режим;
- вынести больше логики в data-driven конфиги;
- расширить cloud save;
- добавить IAP;
- усилить live-ops;
- заменить часть UI-слоя, если это понадобится.
