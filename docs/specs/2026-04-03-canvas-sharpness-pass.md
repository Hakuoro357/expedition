# Canvas Sharpness Pass

## Статус

`Approved`

## Цель

Убрать самое заметное мыло без повторного слома геометрии и ввода.

## Подход

- не менять логические координаты Phaser-сцен;
- не увеличивать внутренний размер игры с последующим remap ввода;
- перевести самые заметные статические canvas-элементы в DOM/SVG overlay.

## Шаг 1. Экран игры

Сделать резкими через overlay:

- верхние слоты `stock / waste / foundations`;
- счётчик колоды;
- пустые foundation-символы.

Canvas остаётся только для:

- самих карт;
- анимаций;
- hit-area верхнего ряда.

## Шаг 2. Главная страница маршрута

Сделать резкими через overlay:

- линии маршрута;
- круги точек;
- цифры точек.

Canvas остаётся только для:

- фоновой подложки;
- hit-area точек;
- свайпа и навигации.

## Шаг 3. Карты

Сделать резче без смены координатной системы:

- ранги и масти на лицевой стороне карт;
- текст и символы в drag-preview;
- текст и символы у flying-card анимаций.
- верхний overlay не должен перекрывать карту в `waste` и `foundation`, если карта там уже лежит.
- если runtime texture-face всё ещё недостаточно резкая, перейти на готовые SVG-лица для всей колоды.

Подход:

- использовать кэшируемые high-resolution texture-face ассеты для лицевой стороны карт;
- не трогать hit-area и раскладку карт.
- целевое финальное решение: полный набор `52` SVG-лиц карт и их прямое использование в Phaser вместо runtime-рендера текста.

## Шаг 4. DOM Cards

Если SVG-лица в Phaser всё ещё недостаточно резкие, перевести сами карты в DOM-слой.

Что меняется:

- face-up карты в раскладе рендерятся как DOM-элементы;
- карты в `waste` и `foundation` тоже рендерятся как DOM-элементы;
- drag-preview рендерится в DOM;
- canvas остаётся только для:
  - фоновой сцены;
  - hit-area;
  - логики drag/drop;
  - анимаций, которые нельзя безопасно вынести сразу.

Требования:

- layout карт и hit-area должны совпасть с текущей сеткой Phaser;
- DOM-карты должны использовать единый SVG/HTML face-template для всей колоды;
- верхний overlay и DOM-карты не должны конфликтовать по z-index;
- переход должен идти поэтапно: сначала статические карты, потом drag-preview, потом анимации.

Риск:

- это заметный рефактор экрана `Игра`, поэтому делать только как отдельный проход с живой browser-проверкой после каждого шага.

### Текущее состояние

Уже сделано:

- верхний ряд `stock / waste / foundations` переведён в DOM/SVG;
- маршрут на главной переведён в DOM/SVG;
- статичные face-up карты в tableau, `waste` и `foundation` переведены в DOM;
- `drag-preview` переведён в DOM;
- угловые индексы и центральные масти карт пересобраны под новый DOM-шаблон;
- анимация открытия карты из `stock` в `waste` переведена на DOM slide + flip/reveal;
- `flying-card` анимации на автодоборе и перелёте в foundation синхронизированы с DOM-card слоем;
- удалён canvas `flyContainer` и `createFlyingCard` метод;
- все тесты проходят (80/80), сборка успешна.

Ещё не завершено:

- требуется финальная ручная проверка в браузере на согласованность DOM-карт и анимаций.

### Обязательное требование на следующий этап

Переход на DOM-карты не должен убирать или ухудшать анимацию добора из `stock`.

Нужно сохранить:

- слайд карты из `stock` в `waste`;
- flip/reveal эффект при открытии карты;
- корректную синхронизацию счётчика `stock`, DOM-карты в `waste` и звука.

## Ограничения

- layout и input не должны менять поведение;
- `Маршрут`, `Игра`, `Архив` и `Detail` должны остаться на текущей координатной сетке;
- если какой-то слой нельзя перевести без риска, он остаётся в canvas до отдельной задачи.
