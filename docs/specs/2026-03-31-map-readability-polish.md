# Map Readability Polish

## Статус

`Draft spec`

## Цель

- Исправить основную проблему экрана карты: плохую читаемость текста и визуальную "рваность" карты.
- Сохранить спокойный экспедиционный тон интерфейса.
- Не переделывать экран целиком и не трогать игровой loop.

## Контекст

По текущему состоянию runtime на экране карты есть три заметные проблемы:

- мелкий текст на canvas плохо читается, особенно кириллица;
- верхний блок перегружен и слишком плотный;
- линии и точки маршрута выглядят "ёлочкой" из-за дробных координат и слишком тонких контуров.

Пользователь выбрал визуальное направление `Serif-Led`:

- крупные заголовки остаются в спокойном serif-стиле;
- утилитарные подписи и номера становятся проще и читаемее;
- карта становится толще и чище, но не более "техничной", чем нужно.

## Решение

### 0. Рендер текста

- Для верхнего блока карты использовать не Phaser `Text`, а внешний `DOM overlay`, привязанный к реальному прямоугольнику `canvas`.
- Для кнопок карты использовать тот же внешний `DOM overlay` только для label-текста.
- Причина:
- браузерный DOM-текст заметно резче и чище, чем canvas-текст Phaser в текущем проекте;
- визуальный референс уже подтверждён в `map-readability-options.html`.
- Важно:
- не использовать для этого `Phaser DOMElement` как основной механизм layout;
- overlay должен якориться к `canvas.getBoundingClientRect()` и масштабироваться от базового размера игры `390x844`.

- В `canvas` оставить:
- фон;
- прогресс-бар;
- маршрут;
- точки маршрута.
- фоны кнопок;
- hover/press states;
- hit areas кнопок.

- В `DOM overlay` перевести:
- title;
- expedition name;
- subtitle;
- chapter header;
- coin display;
- progress text.
- labels кнопок карты.

### 1. Типографика

- Оставить `serif` для:
- названия игры;
- названия главы.

- Перевести в более читаемый UI-стиль:
- название экспедиции;
- subtitle;
- прогресс `0 / 10`;
- номера точек маршрута;
- служебные подписи.

- Увеличить размер мелкого текста на `2-4px` относительно текущего состояния.
- Ослабить зависимость читаемости от `Georgia` на маленьких кеглях.

### 2. Верхний блок

- Сохранить текущий состав:
- title;
- expedition name;
- chapter header;
- progress bar.

- Но изменить иерархию:
- `title` остаётся главным визуальным центром;
- `expedition_name` становится маленьким secondary label;
- `subtitle` можно убирать полностью, если он перегружает шапку и не помогает навигации;
- `chapter header` становится крупнее и проще для быстрого считывания.

- Увеличить вертикальный воздух между этими блоками.

### 3. Карта маршрута

- Скруглить/округлить координаты основных элементов до целых пикселей:
- path points;
- circles;
- текст внутри узлов.

- Увеличить:
- толщину линии маршрута;
- радиусы узлов;
- толщину обводки активной точки;
- размер цифр внутри точек.

- Сохранить существующую логику:
- completed node;
- current node;
- locked node.

- Не менять windowed-отрисовку маршрута и не переделывать структуру progress.

### 4. Вне scope

- не менять narrative content;
- не переделывать Diary/Reward/Game screens в этой задаче;
- не менять art assets;
- не добавлять новые экраны;
- не менять save/data model.
- не переводить на DOM сами кнопочные hit areas и фоновые прямоугольники.

## Файлы

Основной целевой файл:

- `src/scenes/MapScene.ts`

Возможные точечные затронутые файлы:

- `src/scenes/mapSceneOverlay.ts` как helper для HTML overlay;
- `src/ui/canvasOverlay.ts` как reusable helper для привязки overlay к canvas rect;
- `src/ui/createButton.ts` для отделения hit area/background от текстового label;
- `src/services/i18n/locales.ts` только если потребуется упростить слишком длинный subtitle.

## Критерии готовности

- текст на карте читается заметно лучше без приближения;
- кириллица в верхнем блоке больше не выглядит "грязной";
- точки и линии маршрута выглядят ровнее;
- текущая атмосфера игры сохраняется;
- `npm test` и `npm run build` проходят без регрессий.

## Рекомендованный подход реализации

- Делать минимальный targeted polish, а не редизайн.
- Сначала поправить типографику и spacing.
- Затем поправить геометрию маршрута.
- После этого вручную проверить экран карты на desktop/mobile scale.

## Reusable Pattern

- Для экранов с большим количеством текста использовать один и тот же подход:
- Phaser отвечает за графику, интерактив, hit areas и анимации;
- отдельный HTML overlay живёт поверх `canvas` в контейнере приложения;
- overlay позиционируется по реальному rect канваса и масштабируется от логического размера игры;
- если элемент должен быть интерактивным, текст можно держать в `DOM`, а hit area и визуальное состояние в `Phaser`;
- это решение считать базовым кандидатом для `Map`, `Reward`, `Diary`, `Settings`.
